When a client needs updated server state, it needs a strategy for asking: has anything changed yet? A delivery tracker, payment-status page, background export job, or notification panel may all need repeated updates without requiring the user to refresh the page.

Short polling and long polling both use ordinary HTTP requests. The difference is not mainly the endpoint shape; it is how long the server waits before responding and when the client sends the next request.

## Short Answer

**Short polling** means the client sends requests repeatedly on a fixed interval, whether or not anything has changed.

**Long polling** means the client sends a request and the server holds that request open until new data is available or a timeout is reached. After receiving a response, the client immediately opens another request.

| Strategy | Request Behaviour | Best Fit |
| --- | --- | --- |
| Short polling | Client asks every few seconds | Simple systems where delay and extra requests are acceptable |
| Long polling | Server waits before responding | Status changes that are irregular but should appear quickly |
| WebSocket or Server-Sent Events | Persistent server-to-client communication | High-frequency or real-time updates |

The key decision is whether the system benefits from immediate updates enough to justify keeping requests waiting on the server.

## The Core Difference

Assume a delivery order changes status once after two minutes.

With short polling every five seconds, the browser may send about 24 requests before receiving the meaningful update. Most responses simply say that nothing has changed.

```text
Client                  Server
  | --- status? -------> |
  | <--- no change ----- |
  | --- status? -------> |
  | <--- no change ----- |
  | --- status? -------> |
  | <--- delivered ----- |
```

With long polling, the browser sends one request and the server waits. When the order becomes delivered, the server responds immediately.

```text
Client                  Server
  | --- wait status ---> |
  |                      |  Request remains open
  |                      |  Status changes
  | <--- delivered ----- |
  | --- wait status ---> |
```

Both strategies still use HTTP requests and responses. Long polling does not mean the connection stays open forever. Each request normally has a timeout, returns when an event occurs, and is then replaced by a new request.

## How Short Polling Works

In short polling, the frontend starts a timer and repeatedly calls an API endpoint.

For example, a client may request an export job status every five seconds:

```typescript
const jobId = "job_123";

const intervalId = setInterval(async () => {
  const response = await fetch(`/api/jobs/${jobId}/status`);
  const result = await response.json();

  console.log(result.status);

  if (result.status === "completed" || result.status === "failed") {
    clearInterval(intervalId);
  }
}, 5000);
```

The backend endpoint can return the current status immediately:

```typescript
import { Request, Response } from "express";

export function getJobStatus(req: Request, res: Response): void {
  const jobId = req.params.jobId;

  const job = {
    id: jobId,
    status: "processing"
  };

  res.json(job);
}
```

This design is easy to understand because every request is independent. The server receives a request, reads the current state, returns it, and releases the connection.

### Why It Becomes Wasteful

The problem appears when updates are uncommon but clients keep asking frequently.

| Situation | Result |
| --- | --- |
| Status changes once every few minutes | Many responses contain identical data |
| Thousands of active clients poll every few seconds | Request volume grows quickly |
| Poll interval is increased | Server load decreases, but updates appear later |
| Poll interval is decreased | Updates appear faster, but waste increases |

If 10,000 connected clients poll once every five seconds, the server receives approximately 2,000 status requests every second, even when no status has changed.

Short polling therefore creates a trade-off between **update delay** and **unnecessary request volume**.

## How Long Polling Changes the Flow

In long polling, the client still sends a normal HTTP request. The difference is that the server does not respond immediately when no new result exists.

Instead, the server waits for one of two conditions:

1. New data or a status change becomes available.
2. A timeout is reached, so the request returns safely and the client reconnects.

The client logic becomes a repeated request loop rather than a fixed interval:

```typescript
async function listenForJobStatus(jobId: string): Promise<void> {
  while (true) {
    const response = await fetch(`/api/jobs/${jobId}/status/wait`);
    const result = await response.json();

    console.log(result.status);

    if (result.status === "completed" || result.status === "failed") {
      return;
    }
  }
}

listenForJobStatus("job_123");
```

A simplified Express endpoint can hold the response until the job completes or the timeout expires:

```typescript
import { Request, Response } from "express";

const waitingResponses = new Map<string, Response[]>();

export function waitForJobStatus(req: Request, res: Response): void {
  const jobId = req.params.jobId;

  const timeoutId = setTimeout(() => {
    removeWaitingResponse(jobId, res);

    res.json({
      status: "processing",
      timedOut: true
    });
  }, 30000);

  const responses = waitingResponses.get(jobId) ?? [];
  responses.push(res);
  waitingResponses.set(jobId, responses);

  req.on("close", () => {
    clearTimeout(timeoutId);
    removeWaitingResponse(jobId, res);
  });
}

function removeWaitingResponse(jobId: string, res: Response): void {
  const responses = waitingResponses.get(jobId) ?? [];
  const remaining = responses.filter((currentResponse) => currentResponse !== res);

  if (remaining.length === 0) {
    waitingResponses.delete(jobId);
    return;
  }

  waitingResponses.set(jobId, remaining);
}
```

When the background job changes status, the server can respond to every waiting client for that job:

```typescript
import { Response } from "express";

const waitingResponses = new Map<string, Response[]>();

export function publishJobCompletion(jobId: string): void {
  const responses = waitingResponses.get(jobId) ?? [];

  for (const res of responses) {
    res.json({
      status: "completed",
      timedOut: false
    });
  }

  waitingResponses.delete(jobId);
}
```

This example teaches the operating principle, but a production system needs stronger connection cleanup, shared state, authentication, timeout control, and multi-server coordination.

## The Important Client-Side Difference

It is easy to assume that long polling only changes server code because both approaches call an endpoint. In practice, the frontend behaviour also changes.

| Concern | Short Polling Client | Long Polling Client |
| --- | --- | --- |
| Request timing | Sends requests on a timer | Sends the next request after the previous one finishes |
| Waiting behaviour | Waits between requests | Request itself waits on the server |
| Retry handling | Timer naturally triggers the next call | Must reconnect after timeout or failure |
| Duplicate requests | Possible if requests take longer than the interval | Easier to avoid with one active request loop |
| Stopping condition | Clears interval | Exits request loop |

A common short-polling mistake is sending a new request every five seconds even when an earlier request has not returned. This can create overlapping requests under slow network or server conditions.

A long-polling client should normally maintain one outstanding request per watched resource. When it receives a timeout response, it opens a new request. When it receives the final status, it stops.

## Practical Trade-Offs

<CardGrid columns={2}>
  <InfoCard title="Short Polling: Simple">
    The frontend and backend are straightforward. It is often sufficient for low-traffic pages or status checks where a few seconds of delay is acceptable.
  </InfoCard>

  <InfoCard title="Short Polling: Wasteful at Scale">
    Frequent polling can produce large numbers of requests that return unchanged data, especially when updates are rare.
  </InfoCard>

  <InfoCard title="Long Polling: Faster Updates">
    The server can return almost immediately after a status change instead of waiting for the next polling interval.
  </InfoCard>

  <InfoCard title="Long Polling: More Server Responsibility">
    The backend must manage waiting requests, timeouts, client disconnects, deployment topology, and resource limits.
  </InfoCard>
</CardGrid>

Long polling usually reduces repeated empty responses, but it does not make connections free. A waiting request still consumes some server and infrastructure capacity.

For Node.js, waiting on network or event completion is generally compatible with its event-driven model because the process does not need to block a thread while waiting. However, application memory, open sockets, load balancer timeouts, reverse proxy configuration, and database usage still matter.

## Real Production Concerns

A basic long-polling example can work locally and still fail after deployment. The design must account for the surrounding infrastructure.

| Concern | Why It Matters | Practical Handling |
| --- | --- | --- |
| Request timeout | Proxies may close idle requests | Return before infrastructure timeout and reconnect |
| Client disconnect | Closed browser tabs leave waiting work behind | Remove response listeners on connection close |
| Multiple backend instances | Waiting request may be on one server while update occurs on another | Publish events through Redis Pub/Sub, a queue, or shared event system |
| Authentication | A client must not observe another user's job | Authorize every wait request |
| Too many open requests | Memory and socket usage can increase | Limit connections and monitor active waits |
| Failure and retries | Networks disconnect normally | Use bounded retry with delay or backoff |

Suppose the application has two backend instances:

```text
Client waits on Server A
Background worker updates status through Server B
```

If Server B only stores the event in its own application memory, Server A does not know that it should answer the waiting client. In a scaled deployment, long polling often needs a shared event mechanism rather than an in-memory map alone.

## When to Use Each Strategy

Use **short polling** when:

- The feature is simple and traffic is low.
- Updates do not need to appear instantly.
- The status is checked only for a short period, such as waiting for one payment confirmation.
- Operational simplicity matters more than request efficiency.

Use **long polling** when:

- Updates happen unpredictably.
- The user should see changes quickly.
- Repeated unchanged responses would create significant waste.
- WebSockets or Server-Sent Events are not suitable for the environment.

Consider **Server-Sent Events** when the server frequently pushes one-way updates to the browser, such as notifications or streaming job progress.

Consider **WebSockets** when both client and server need frequent, low-latency messages, such as chat, live collaboration, or multiplayer interaction.

| Requirement | Suitable Starting Choice |
| --- | --- |
| Refresh status every 30 seconds on a small admin page | Short polling |
| Wait for an asynchronous report to finish | Short polling or long polling |
| Show delivery-status changes quickly to many users | Long polling or Server-Sent Events |
| Stream continuous server updates to the browser | Server-Sent Events |
| Two-way real-time communication | WebSockets |

Do not choose long polling only because it sounds more advanced. It is valuable when the request reduction or responsiveness improvement is meaningful.

## Common Mistakes

### Mistake 1: Thinking Long Polling Uses Only One Request

Long polling still creates repeated HTTP requests. The difference is that each request waits longer and is more likely to return useful information.

### Mistake 2: Holding Database Connections Open

A waiting HTTP request should not normally keep a database transaction or database connection occupied for the entire wait period. Read the initial status, wait for an application event, and query again only when necessary.

### Mistake 3: Ignoring Timeouts

A server should not keep a request open indefinitely. Timeouts allow the client to reconnect, allow infrastructure to behave predictably, and prevent abandoned waits from accumulating.

### Mistake 4: Forgetting Cleanup

When the user closes the tab, navigates away, or loses network access, the server must remove that waiting response from its internal tracking.

### Mistake 5: Using In-Memory Events After Scaling Horizontally

An in-memory listener can work with one Node.js process. Once requests and updates can reach different instances, the event must be distributed through shared infrastructure.

## A Practical Implementation Order

Build the feature in stages instead of immediately introducing a complex real-time architecture.

1. Begin with short polling using a reasonable interval, such as five or ten seconds.
2. Measure how often clients poll, how often responses actually change, and how much delay users experience.
3. Move to long polling when unchanged responses are frequent or users need faster updates.
4. Add request timeouts, disconnect cleanup, authorization, and monitoring.
5. Introduce shared event delivery when the backend runs across multiple instances.
6. Move to Server-Sent Events or WebSockets only when the communication pattern requires them.

This order keeps the design proportional to the actual problem. A simple status page does not automatically require a persistent real-time channel.

## The Main Principle

Short polling asks repeatedly whether something has changed. Long polling waits until something changes before answering.

The better strategy depends on request volume, acceptable update delay, deployment complexity, and whether the system can safely manage waiting connections. Start with the simplest design that meets the user experience requirement, then change the transport only when measurement shows that the current approach is producing unnecessary work or unacceptable delay.
